# FoodFast Order Management UI — Business Specification

## Overview

A single-page web application that allows users to create, view, delete, and calculate delivery fees for food delivery orders. The frontend communicates with the existing FoodFast REST API (built in week 8) and displays results in real time. This UI is the surface that End-to-End tests will verify.

## Data Model

### CreateOrderForm (user input)

| Field | Type | UI Element | Validation |
|---|---|---|---|
| Customer Name | string | Text input | Required, non-empty (enforced by API) |
| Cart Subtotal | decimal | Number input | Must be >= 0 (enforced by API) |
| Distance (km) | double | Number input | Must be > 0 and <= 100 (enforced by API) |
| Rush Hour | boolean | Checkbox | Optional, defaults to false |

> **Note:** ID is auto-generated by the database. It is not part of the form — it only appears in the table after creation.

### OrderRow (displayed in table)

| Field | Type | Source | Display Format |
|---|---|---|---|
| ID | integer | API response `id` | Auto-generated, plain number |
| Customer Name | string | API response `customerName` | Plain text |
| Cart Subtotal | decimal | API response `cartSubtotal` | `$X.XX` |
| Distance | double | API response `distanceInKm` | `X km` |
| Rush Hour | boolean | API response `isRushHour` | `Yes` / `No` |
| Delivery Fee | decimal | API response `deliveryFee` | `$X.XX` |
| Created At | datetime | API response `createdAt` | Date string |

### FeeBreakdown (displayed on calculate)

| Field | Type | Source | Display Format |
|---|---|---|---|
| Order ID | integer | API response `orderId` | Plain number |
| Customer Name | string | API response `customerName` | Plain text |
| Cart Subtotal | decimal | API response `cartSubtotal` | `$X.XX` |
| Distance | double | API response `distanceInKm` | `X km` |
| Rush Hour | boolean | API response `isRushHour` | `Yes` / `No` |
| Delivery Fee | decimal | API response `deliveryFee` | `$X.XX` |
| Total | decimal | API response `total` | `$X.XX` |

## Business Rules

### 1. Create Order

- User fills in customer name, cart subtotal, distance, and optionally checks rush hour
- On submit, the frontend sends `POST /api/orders` to the API
- **Success (201):** The order appears in the table immediately (table refreshes)
- **Validation error (400):** The frontend displays the error message(s) returned by the API
- The form clears after a successful submission

### 2. List Orders

- On page load, the frontend fetches `GET /api/orders` and displays all orders in a table
- Orders are sorted by creation date (newest first, as returned by the API)
- **Empty state:** When no orders exist, display "No orders yet" instead of an empty table

### 3. Delete Order

- Each order row has a Delete button
- On click, the frontend sends `DELETE /api/orders/{id}` to the API
- **Success (204):** The order disappears from the table immediately (table refreshes)
- **Not found (404):** Should not happen in normal flow (order was visible)

### 4. Calculate Delivery Fee

- Each order row has a Calculate Fee button
- On click, the frontend sends `POST /api/orders/{id}/calculate-fee` to the API
- **Success (200):** The fee breakdown is displayed (inline or modal)
- The breakdown shows: cart subtotal, distance, rush hour status, delivery fee, and total

### 5. Data Persistence

- All data is stored in the API's SQLite database
- Page refresh should show the same orders (data is not in-memory React state)

## API Endpoints (Backend Contract)

The frontend calls these existing endpoints (from week 8):

| Method | Endpoint | Request Body | Success Response | Error Response |
|---|---|---|---|---|
| POST | `/api/orders` | `{ customerName, cartSubtotal, distanceInKm, isRushHour }` | `201` + order JSON | `400` + `{ errors: [...] }` |
| GET | `/api/orders` | — | `200` + order array | — |
| GET | `/api/orders/{id}` | — | `200` + order JSON | `404` |
| DELETE | `/api/orders/{id}` | — | `204` | `404` |
| POST | `/api/orders/{id}/calculate-fee` | — | `200` + fee breakdown JSON | `404` |

## Delivery Fee Rules (Reference)

The fee is calculated by the backend's `DeliveryPricingEngine`. These rules are relevant for verifying UI displays:

| Distance | Base Fee | Rush Hour | Free Delivery |
|---|---|---|---|
| < 5 km | $2.00 | x 1.5 = $3.00 | Cart >= $50: $0.00 |
| 5–10 km | $5.00 | x 1.5 = $7.50 | Cart >= $50: $0.00 |
| >= 10 km | $10.00 | x 1.5 = $15.00 | Cart >= $50: $0.00 |

## Example Scenarios

### Happy Path

| Scenario | Input | Expected UI Result |
|---|---|---|
| Standard order | "Alice", $30, 6 km, non-rush | Order appears in table with $5.00 fee |
| Rush hour order | "Bob", $20, 3 km, rush | Order appears with $3.00 fee |
| Free delivery | "Charlie", $60, 12 km, non-rush | Order appears with $0.00 fee |
| Delete order | Click delete on existing order | Order disappears from table |
| Calculate fee | Click calculate on standard order | Breakdown shows: fee $5.00, total $35.00 |
| Empty state | No orders in DB | "No orders yet" message displayed |
| Page refresh | Create order, refresh browser | Order still visible after refresh |

### Sad Path

| Scenario | Input | Expected UI Result |
|---|---|---|
| Empty customer name | "", $30, 6 km | Error message: "CustomerName is required" |
| Negative subtotal | "Alice", -$10, 6 km | Error message: "CartSubtotal must be >= 0" |
| Distance too far | "Alice", $30, 150 km | Error message: "DistanceInKm must be <= 100" |
| Zero distance | "Alice", $30, 0 km | Error message: "DistanceInKm must be > 0" |
| API server down | Any input, API not running | Error message displayed (not a crash) |

## UI Component Structure

```
App
├── OrderForm          — form with 4 inputs + submit button
├── OrderTable         — table of all orders
│   └── OrderRow       — one row per order (delete + calculate buttons)
└── FeeBreakdown       — fee calculation result display
```
